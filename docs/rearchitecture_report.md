# HebMorph Rearchitecture and Migration Report

## 1. The Gold (Core Logic & Data)
The real value of HebMorph, stripped of its Lucene and Elasticsearch wrappers, lies in its custom Hebrew morphological analysis logic and data structures:

*   **Morphological Dictionary and Tree (DictRadix):** The library implements a custom Radix tree (`DictRadix`) tailored for morphological lookups. It maps Hebrew words to their base lemmas and morphological data.
*   **HSpell Integration:** The `HSpellLoader` is the heart of the dictionary. It loads binary `hspell-data-files` (like `hebrew.wgz` and prefixes/suffixes files) directly into the `DictHebMorph` structure. This represents the curated knowledge of Hebrew linguistics.
*   **Lemmatizer Engine (`Lemmatizer.java` / `StreamLemmatizer.java`):** This applies rules to handle prefixes (e.g., trying to strip prefixes and doing dictionary lookups) and tolerances.
*   **Tolerance and Normalization:** The `LookupTolerators` handles "Ktiv Male/Ktiv Haser" issues (e.g., tolerating extra or missing 'Vav' or 'Yud' characters), ensuring high recall during text analysis.
*   **Hebrew-Specific Tokenization Rules (`Tokenizer.java`):** Custom rules for handling Geresh (`'`), Gershayim (`"`), and Makaf (`-`), and filtering non-Hebrew characters.
*   **Linguistic Constants:** `HebrewUtils.java` and `HebrewCharacters.java` hold constants critical for accurately interpreting Hebrew text according to linguistic academy standards.

## 2. Project Overview
HebMorph currently functions primarily as an open-source Lucene and Elasticsearch analyzer, designed to make Hebrew properly searchable with high recall and precision.
*   **Current State:** It is heavily coupled with `org.apache.lucene` and specific versions of the Lucene/Solr/Elasticsearch stack.
*   **Language Implementations:** The core is written in Java (compatible up to Lucene 6.6.0), with an older, slightly buggy .NET port targeting `Lucene.NET` 3.0.3.
*   **Data Source:** Relies fundamentally on data generated by Hspell (v1.2).
*   **Use Cases:** Traditionally embedded directly in search engines, but structurally it contains the components to act as a standalone text processing application.

## 3. Missing Parts & Gaps
The project has not been maintained for several years, leading to several significant gaps:
*   **Outdated Tech Stack:** The Java versions explicitly support up to Lucene 6.6 / Elasticsearch 5.x. Modern Elasticsearch is on 8.x, and Lucene is on 9.x. The dependency chain is broken for modern search infrastructures.
*   **Unresolved TODOs:** Dozens of TODOs are scattered throughout the core logic, such as:
    *   Stop word filtering based on morphological data needs implementation.
    *   Gematria handling is incomplete.
    *   Better abbreviation and acronym handling.
    *   Following the Academy's punctuation rules strictly is marked as "TODO".
*   **Incomplete Ports:** The .NET version lacks bug fixes that were applied to the Java version.
*   **Memory Efficiency:** The `DictRadix` implementation, while fast, can be extremely memory-hungry compared to modern Finite State Transducers (FSTs) used in modern Lucene.

## 4. Migration Plan
To reimagine HebMorph as a standalone preprocessing app and modern search plugin, the following migration plan is recommended:

1.  **Extraction and Decoupling:** Extract the core logic (`com.code972.hebmorph.*`) entirely from the Lucene packages (`org.apache.lucene.*`). The core must have zero dependencies on search engines.
2.  **Standalone API Creation:** Expose a clean interface, e.g., `HebrewAnalyzer.analyze(String text) -> List<Token>`.
3.  **Data Structure Modernization:** Replace `DictRadix` with a highly optimized Finite State Transducer (FST) or Double-Array Trie to reduce memory footprint and improve load times.
4.  **CLI / Standalone App:** Build a thin CLI/Daemon layer over the core API that can take JSON/Text inputs and return analyzed tokens, serving as a preprocessing stage *before* Elasticsearch.
5.  **Modern Plugin Wrappers:** Rewrite the Elasticsearch and Solr plugins targeting modern versions (ES 8.x, OpenSearch 2.x) using the new standalone core as a pure dependency.

## 5. Suggestions for Improvement
*   **Data Update:** HSpell v1.4 is available; the library currently uses data from v1.2. The loading mechanism should be updated or generalized to handle newer HSpell dictionaries.
*   **Language Upgrade:** Rewrite the core in a highly performant, memory-safe, and easily distributable language like Rust, Go, or modern Java (17+). Rust/Go would make it exceptionally easy to distribute as a standalone preprocessor CLI without JVM overhead.
*   **Configuration Flexibility:** Allow passing dictionaries via standard file formats (like JSON/CSV) alongside HSpell binaries to easily allow users to add custom domains (e.g., legal or medical Hebrew terms).

## 6. Recommendations: Rewrite vs Port
**Recommendation: Complete Rewrite with Splitting Strategy.**

Given the age of the codebase, the technical debt (TODOs), and the tight coupling to legacy Lucene versions, a complete rewrite is the most prudent path forward.

**The Split Architecture:**
1.  **`hebmorph-core` (The Standalone App):** Rewrite the core morphological engine in Rust or Go. This creates a blazing-fast, zero-dependency binary that can be used via CLI, HTTP API, or as a C-binding (FFI) for other languages. This fulfills the desire for a standalone app to process text *before* it goes to Elastic.
2.  **`hebmorph-elasticsearch` / `hebmorph-opensearch`:** Build lightweight plugins in Java that simply wrap the core engine (or communicate with it) to provide native analyzer support for those who still want plugin-based integration.
3.  **`hebmorph-dictionary`:** Isolate the dictionary compilation scripts so updates to HSpell can be independently published and consumed by the core engine.

A rewrite allows you to implement modern data structures (FSTs) from day one, drastically cutting memory usage while preserving the exact "gold" linguistic rules that made HebMorph great.
