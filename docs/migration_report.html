HebMorph Migration & Rearchitecture Report

Status: The core logic of HebMorph has been successfully extracted from Java/Elasticsearch and rewritten into a decoupled, standalone C# (.NET 10) architecture.

1. The "Split-Way" Architecture

To ensure components can be used independently (e.g., just the Tokenizer, or just the Lemmatizer) and are Elasticsearch-agnostic, the project was split into several distinct C# Class Libraries under the new HebMorphCsharp root folder:

HebMorph.Core

Contains the absolute linguistic truth: constants, character mapping (Niqqud, Geresh, Makaf), and shared interfaces (like MorphData and MorphType). It has zero external dependencies.

HebMorph.Dictionary

Handles the loading and querying of the morphological dictionary. It implements a clean IMorphologicalDictionary interface.

  • Extendable Data: The interface abstracts the backend. Currently, an FstDictionary wrapper has been provided to prepare for Lucene.NET 4.8 FST implementations.
  • Replacable: You can inject custom logic (like reading JSON/CSV files) to supplement or replace the binary HSpell wgz files.

HebMorph.Tokenizer

A pure C# Hebrew Tokenizer that respects academy rules regarding punctuation. It operates solely on strings and outputs custom Token objects, entirely separate from Lucene's TokenStream.

HebMorph.Lemmatizer

The brain of the operation. It takes tokens and resolves them to their base lemmas, applying tolerances (Ktiv Male/Haser) and stripping prefixes.

HebMorph.WorkerService

A standalone .NET Worker Service demonstrating how to run the analyzers as a background daemon (e.g., listening to a queue and preprocessing text before it is sent to Elastic/OpenSearch).

2. Addressed TODOs & Business Rules

During the migration, several legacy TODOs in the Java codebase were resolved using standard Hebrew linguistic rules:

  • Tokenizer: Makaf Shortening (י-ם) - Makafs are now retained inside tokens if they are surrounded by valid Hebrew letters, allowing the Lemmatizer to see the exact abbreviation instead of splitting it into "י" and "ם".
  • Tokenizer: Abbreviations (פרופ') - The Geresh/Gershayim characters are now parsed and kept inside the token boundaries, ensuring acronyms are passed intact to the Lemmatizer.
  • Lemmatizer: Stop Words - Implemented stop word filtering during lemmatization. It flags tokens as IsStopWord = true and filters them out of the valid lemma list to prevent noise.
  • Lemmatizer: Gematria Test - Implemented GematriaCalculator. The Lemmatizer detects valid Gematria structures (e.g., תשפ"ג) and assigns them an integer value rather than attempting to morphologically analyze them as standard nouns.
  • Lemmatizer: Scoring/Confidence - Implemented a basic scoring mechanism (CalculateScore) that prefers exact morphological matches over prefix-stripped or Ktiv-tolerated matches.
  • Lemmatizer: Unknown Nouns - If a word yields zero lemmas from the dictionary (and isn't a Gematria/Stopword), the original token is explicitly added back as a default lemma (with a 0 mask) to preserve it for downstream search, assuming it is a proper noun/name.

3. Future Steps & FST Dilemma

We chose to use the modern Lucene.Net 4.8.0-beta for the FST data structures as requested. Because highly compressed FSTs in C# require tight control over primitive byte arrays, the FstDictionary class currently mocks the dictionary internally to prove the architectural flow. The next technical step would be porting the binary stream reader from HSpellLoader.java to feed directly into the Lucene.Net.Util.Fst.Builder.

Because the dictionary is behind the IMorphologicalDictionary interface, this future FST optimization can be done without altering a single line of the Tokenizer, Lemmatizer, or Worker Service.
